# HX35HM SO101 IK 改 FJT 统一执行说明

## 1. 文档目的

这份文档专门解释一个关键架构改动：

**为什么要把原来 `/go_to_pose` 的 IK 流式执行，改成“生成整条 `JointTrajectory` 后统一交给 `FollowJointTrajectory` 执行”。**

这不是普通调参，而是抓取控制链里一个很重要的执行架构调整。

这份文档重点回答：

- 原来 `/go_to_pose` 是怎么执行的
- 它为什么会带来平滑度和一致性问题
- 为什么我们没有简单地“全都交给 MoveIt”
- 这次到底改了哪些代码
- 改完以后实际改善了什么
- 现在这条链还有哪些边界和后续方向

相关文档：

- `HX35HM_SO101_机械臂控制链详解.md`
- `HX35HM_SO101_红球抓取调试日志.md`
- `HX35HM_SO101_控制链接口速查表.md`

## 2. 改动前的控制结构

在这次改动之前，你的机械臂手臂部分其实长期存在两条执行链：

### 2.1 前半段：IK / 自定义轨迹链

抓取前半段，也就是：

- `hover_high`
- `pregrasp`
- `grasp`
- `retreat`

大多数情况下走的是：

- `/go_to_pose`

这条链由 `so101_kinematics/cartesian_motion_node.py` 负责。

它原来的执行逻辑是：

1. 接收目标 `Pose`
2. 调用 `motion_planner.py` 生成 `(ts, qs)` 轨迹
3. 交给 `TrajectoryExecutor`
4. 再通过 timer 以 `50Hz` 方式不断发布：
   - `/follower/forward_controller/commands`

也就是说，**它不是一次性把完整轨迹交给控制器执行，而是在本节点里做时间采样，再把每一拍的关节目标流式发出去。**

### 2.2 后半段：MoveIt / FJT 链

抓取完成后回 `rest`，主路线大多走：

- MoveIt named target

这条链的执行逻辑是：

1. MoveIt 规划完整 `JointTrajectory`
2. 发到：
   - `/follower/arm_trajectory_controller/follow_joint_trajectory`
3. 由 `hx35hm_bridge` 执行

也就是说，**后半段已经是标准 FJT 轨迹执行语义。**

### 2.3 夹爪是第三条独立链

夹爪一直单独走：

- `/follower/gripper_controller/gripper_cmd`

这条链本身没有问题，也不应该并入手臂轨迹链。

所以真正的问题，不是“有第三条夹爪链”，而是：

**手臂本身前后用了两套完全不同的执行后端。**

## 3. 改动前的明显缺陷

这个结构能跑，也能抓，但有几个长期存在的明显缺陷。

## 3.1 手臂前后执行风格不一致

前半段是：

- IK 解目标
- 本地轨迹采样
- 50Hz 流式关节命令

后半段是：

- MoveIt 完整规划
- 标准 `JointTrajectory`
- FJT action 执行

这会带来：

- 动作节奏不一致
- 成功/失败语义不一致
- 平滑性的来源不一致
- 前半段像在“追目标”
- 后半段像在“执行轨迹”

结果就是：

- 整体动作容易有段落感
- 回 `rest` 和前段靠近目标看起来像两台不同风格的机器人

## 3.2 50Hz 流式关节命令对当前硬件不够友好

`/follower/forward_controller/commands` 本质上是：

- 当前这一刻关节该去哪里

它不是：

- 完整轨迹
- 连续速度约束
- 连续加速度约束

再考虑到底层控制板实际语义是：

- `位置 + duration`

就会出现一个典型现象：

- 上游每 `20ms` 给一个新目标
- 底层不断重定目标
- 精度可以做高
- 但观感容易机械、硬、碎

## 3.3 很难把“平滑”和“精度”同时做好

当时反复遇到的现象是：

- 想让动作更柔，就得把底层时间常数调大
- 但一调大，前半段跟随又开始变差
- 想把抓取精度拉高，就得让动作更跟手
- 但一更跟手，机械感又会变强

这说明问题已经不只是参数了，而是**执行结构本身不够统一。**

## 3.4 调试复杂度很高

同样是“前半段动作不好看”，你当时得分辨到底是：

- IK 解有问题
- `TrajectoryExecutor` 有问题
- 流式采样频率不合适
- `stream_command_duration` 不合适
- `bridge` 不合适

这是典型的结构复杂度过高问题。

---

## 4. 为什么不直接“全部交给 MoveIt”

这个问题当时其实认真考虑过。

表面上看，既然后半段已经走 MoveIt/FJT，那似乎可以把前半段也全部交给 MoveIt。

但这条路没有直接采用，原因也很明确。

## 4.1 MoveIt 更擅长完整规划，不一定更擅长最后几厘米的任务动作

对于：

- `hover_high`
- `pregrasp`
- `grasp`

这些段，你希望的是：

- 靠近方式可控
- 最后几厘米足够稳
- 姿态和路径都更贴任务需求

而不是只是：

- “规划出一条能到的轨迹”

MoveIt 更擅长：

- 全局路径
- 碰撞场景
- 完整轨迹规划

但对于最后靠近目标的那几厘米，它不天然比一条有任务含义的 IK 轨迹更合适。

## 4.2 你当前抓取前半段的精度其实主要来自 IK 这条链

前半段后来之所以精度能做到很高，一个重要原因就是：

- `go_to_pose` 这条链更贴目标
- 更容易围绕抓取姿态和接近方式做工程化控制

所以，完全丢掉 IK 并不是最优路线。

## 4.3 真正的问题不是“有 IK”，而是“IK 直接流式执行”

我们后面逐步看清楚的是：

- IK 负责生成目标和轨迹，这件事没有问题
- 真正的问题在于它后面接的是：
  - `TrajectoryExecutor + 50Hz forward_controller/commands`

所以最优方向不是：

- “前半段全部换成 MoveIt”

而是：

- **保留 IK 负责算目标和轨迹**
- **但把执行后端统一成 FJT**

这就是后来真正落地的方案。

---

## 5. 这次改动的核心思路

核心思路可以用一句话概括：

**IK 继续负责“算怎么去”，FJT 统一负责“怎么执行”。**

也就是把原来的：

1. `/go_to_pose`
2. 生成 `(ts, qs)`
3. `TrajectoryExecutor`
4. timer 50Hz 发布 `/forward_controller/commands`

改成：

1. `/go_to_pose`
2. 生成 `(ts, qs)`
3. 构造成完整 `JointTrajectory`
4. 通过 `FollowJointTrajectory` action 发送给控制层
5. 由 `hx35hm_bridge` 统一执行

这就把手臂执行后端统一了。

---

## 6. 具体改了什么

这次改动主要集中在：

- `so101_kinematics/cartesian_motion_node.py`

## 6.1 加入 FJT action client

新增：

- `ActionClient(FollowJointTrajectory, ...)`

并新增参数：

- `fjt_action_name`
- `fjt_server_wait_timeout_s`
- `fjt_result_timeout_padding_s`

默认 action 名称是：

- `/follower/arm_trajectory_controller/follow_joint_trajectory`

## 6.2 `/go_to_pose` 不再本地流式执行

旧逻辑是：

- `(ts, qs)` -> `TrajectoryExecutor` -> `forward_controller`

新逻辑是：

- `(ts, qs)` -> `JointTrajectory` -> `FollowJointTrajectory`

也就是：

- 不再让 `cartesian_motion_node` 自己 50Hz 负责整条服务轨迹执行
- 而是把完整轨迹交给 controller action

## 6.3 `/go_to_joints` 也统一改成 FJT

这一步很重要。

因为如果只改 `/go_to_pose`，`/go_to_joints` 还保留旧方式，那统一仍然不完整。

所以这次是：

- `/go_to_pose` -> FJT
- `/go_to_joints` -> FJT

统一完成。

## 6.4 `servo_target` 保留原样

这次没有去动：

- `/servo_target`

原因是：

- 它本来就是实时小步交互式控制
- 和服务驱动的完整轨迹语义不同
- 可以继续保留为交互调试链

所以当前状态是：

- 服务驱动动作：统一走 FJT
- 实时交互 servo：保留原语义

这也是比较合理的分层。

---

## 7. 这次改动带来的直接收益

## 7.1 手臂执行后端终于统一了

现在手臂主路线已经变成：

- 前半段 IK 服务 -> FJT
- 后半段 MoveIt -> FJT

也就是说，**不再存在“前半段流控、后半段轨迹 action”这种明显割裂。**

## 7.2 调试问题更聚焦

改之前，如果前半段不顺，很容易怀疑：

- timer
- 采样频率
- 流式 duration
- forward_controller

改之后，问题开始更聚焦到：

- IK 生成的轨迹是不是合理
- 默认抓取姿态是不是合理
- 底层 FJT 重采样是不是合理

这其实是好事，因为问题边界更清楚了。

## 7.3 为后面发现真正瓶颈创造了条件

统一 FJT 之后，很快暴露出来的真正问题不是“流式执行不平滑”，而是：

- 默认抓取姿态把 `wrist_roll` 推到了很差的 IK 分支
- 某些姿态接近关节边界

这说明：

- 之前流式执行确实掩盖了一部分真实问题
- 统一后端后，系统终于开始把真正的几何和姿态问题暴露出来

这也是为什么后来你能把 `wrist_roll` 限位问题真的查清楚。

---

## 8. 这次改动没有解决什么

这次改动很重要，但它不是万能药。

## 8.1 它没有让底层板子变成工业伺服

当前控制板和总线舵机的语义还是：

- 位置 + duration

所以即便统一成 FJT，也只是：

- 执行链更统一
- 不代表立刻拥有工业机械臂那种丝滑连续控制能力

## 8.2 它没有替你自动选好抓取姿态

统一 FJT 后端之后，`wrist_roll` 限位问题反而更明显地暴露出来了。

这也证明：

- 执行后端统一了
- 但默认抓取姿态仍然可能很糟

所以后续还必须继续做：

- 姿态搜索
- wrist-roll 友好 seed 选择
- 默认抓取四元数修正

## 8.3 它没有自动解决视觉误差

前半段轨迹再统一，如果：

- RGB-D 外参不准
- 视觉目标点本身有偏

那动作还是会不理想。

所以这一步只是把执行链问题收敛了，不是把整个系统所有问题都解决了。

---

## 9. 为什么这一步对你现在的工程非常关键

因为你现在项目的主要目标不是“单次能抓到”，而是：

- 能抓
- 精度高
- 动作尽量顺
- 结构上后续还能维护

如果继续保留老结构：

- 前半段一种执行器
- 后半段另一种执行器

那后面每次调平滑度、调姿态、调桌面、调视觉，你都要同时面对两套完全不同的执行语义。

这会越来越难维护。

而统一到 FJT 以后，至少手臂执行层的主语义已经更清楚了：

- IK / MoveIt 负责生成轨迹
- FJT / bridge 负责执行轨迹

这个边界是很重要的。

---

## 10. 当前架构状态

截至当前版本，你的手臂执行结构可以这样理解：

### 10.1 前半段

- `hover_high / pregrasp / grasp / retreat`
- 主要仍由 `/go_to_pose` 驱动
- 但执行后端已经统一成 FJT

也就是：

- 仍然保留 IK 的任务可控性
- 但不再保留旧的 50Hz 流式服务执行方式

### 10.2 后半段

- `return rest`
- 仍然主要由 MoveIt named target 负责
- 本来就是 FJT

### 10.3 夹爪

- 继续独立走：
  - `/follower/gripper_controller/gripper_cmd`

不并入手臂主轨迹链。

这套结构目前是合理的。

---

## 11. 当前方案的优点

可以直接总结成 5 点：

1. 手臂执行后端统一，结构更清楚
2. 保留了 IK 对抓取前半段的任务可控性
3. 避免了旧的 50Hz 流式服务执行风格
4. 更适合当前 `hx35hm_bridge + 控制板` 这套硬件链
5. 为后续调姿态、调视觉、调平滑度提供了更干净的基础

---

## 12. 当前方案的局限

这套方案是当前工程上的更优解，但也仍然有边界。

1. 前半段和后半段虽然执行后端统一了，但轨迹来源仍不同：
   - 前半段来自 IK / 自定义 planner
   - 后半段来自 MoveIt
2. 默认抓取姿态如果不合理，依然会把机械臂推到差的关节分支
3. 底层协议仍不是高阶连续伺服
4. 如果视觉目标本身有偏，FJT 也不会替你修正目标

所以这一步是**非常关键的架构优化**，但它仍然只是整条系统收敛过程中的一环。

---

## 13. 最终结论

把 `/go_to_pose` 从“IK + 本地流式执行”改成“IK 生成轨迹 + FJT 统一执行”，核心价值不在于一句“改成 MoveIt 了”。

更准确的说法是：

**不是把 IK 改没了，而是把 IK 生成轨迹后的执行方式，改成了和 MoveIt 同一类的标准轨迹执行语义。**

这一步解决的核心问题是：

- 手臂执行后端长期不统一
- 流式执行和 FJT 混用
- 调试边界不清
- 平滑度和精度长期互相拉扯

它没有替代 IK，而是让 IK 这条链终于能以更适合当前工程的方式落地执行。

---

## 14. 后续调试中发现的关键修正：四元数混杂因素

在这份文档最初形成后，后续现场调试又暴露出一个非常关键的混杂因素：

- 不是只有执行后端从“流式 IK”切到了“FJT”
- **固定抓取四元数也曾被改过**

这件事一开始没有被完全隔离出来，导致当时对“IK 改 FJT 后不准”的判断被放大了。

### 14.1 远端参考仓库与本地异常版本的差异

对比后确认，远端参考仓库默认抓取姿态是：

- `qx = 0.0`
- `qy = 1.0`
- `qz = 0.0`
- `qw = 0.0`

而本地一度出现过另一组固定四元数：

- `qx = 0.46467987`
- `qy = 0.76828305`
- `qz = 0.14769257`
- `qw = 0.41472966`

在当前抓取节点里，默认又是：

- `use_marker_orientation = false`
- `use_rpy = false`

所以末端抓取姿态会**完全吃固定四元数**。

这意味着：

- `wrist_roll` 会被带到另一条关节分支
- `wrist_flex` 的解会明显变化
- 即便目标点一样，接近球的姿态也会完全不同

### 14.2 为什么这会污染对 FJT 的评价

如果同时发生下面两件事：

1. 执行后端从流式 IK 切到 FJT
2. 固定抓取姿态四元数也换掉

那么现场看到的异常就不再只是：

- 到位误差

而会变成：

- `wrist_roll` 也开始做以前没有的动作
- 夹爪姿态整体看起来就不对
- 抓取失败会被误以为都是 FJT 的问题

也就是说，当时那轮实验其实同时改了两个变量：

- 执行方式
- 末端固定姿态

这会让结论不够干净。

### 14.3 修正后再次做 A/B 测试

在把四元数恢复成正常值：

- `qx=0`
- `qy=1`
- `qz=0`
- `qw=0`

之后，又重新做了一轮“只切执行后端，不再改姿态”的 A/B 测试。

结果表明：

- 四元数错误确实显著放大了之前对 FJT 的负面感受
- 但即使四元数恢复正确，执行后端不同仍然会带来精度差异

因此更准确的结论不是：

- “之前所有不准都只是四元数的问题”

而是：

- **之前的不准，既受 FJT 影响，也受错误固定四元数影响**
- **其中四元数错误是一个重要混杂因素**

### 14.4 当前最终保留的姿态参数

当前工作区已经固定回正常抓取四元数：

- `qx = 0.0`
- `qy = 1.0`
- `qz = 0.0`
- `qw = 0.0`

后续再评估任何执行链差异时，都应保持这组四元数不变，否则实验结论仍然会被污染。

## 15. `/go_to_pose` 段间停顿的原因与优化

后续实机调试里还发现一个体验问题：

- 从一个 `/go_to_pose` 目标切到下一个目标时
- 会有明显“停一下再动”的感觉

这个问题后来也单独排查并做了优化。

### 15.1 停顿不是只有一个来源

最终确认停顿主要来自两层：

#### A. `/go_to_pose` 服务内部的到位收敛等待

`cartesian_motion_node.py` 在轨迹执行完以后，并不会立即返回。

它还会：

1. 读取当前末端位姿
2. 计算与目标位姿的误差
3. 在一个时间窗口内继续等待是否收敛

也就是说，除了“轨迹本身执行时间”，还存在一段：

- “执行完以后继续等到位”的隐藏等待

这层等待之前默认较长，是阶段切换停顿感的主因之一。

#### B. 抓取节点里的显式 `settle`

在 `so101_visual_grasp.cpp` 里，`hover_high / pregrasp / grasp / retreat`
之间还会额外插入短暂 `sleep`：

- `ik_pregrasp_settle_s`
- `ik_grasp_settle_s`
- `ik_retreat_settle_s`

这些值虽然比主执行时间小很多，但也会在视觉上形成“每一步都顿一下”的感觉。

### 15.2 做过的优化

后续已经对默认值做过一轮压缩：

- `goal_wait_timeout_s`
  - 从较长等待缩短为更激进的 `0.35s`
- `ik_pregrasp_settle_s`
  - 压到 `0.03s`
- `ik_grasp_settle_s`
  - 压到 `0.02s`
- `ik_retreat_settle_s`
  - 压到 `0.02s`

这轮优化的原则不是“完全去掉所有等待”，而是：

- 保留最基本的收敛和稳定时间
- 但尽量消除肉眼可见的卡顿

### 15.3 为什么没有直接把等待全删掉

因为这套机械臂不是工业级高刚性伺服，真实系统存在：

- 背隙
- 柔性
- 总线位置命令执行误差

如果把等待完全删掉，常见风险是：

- 视觉上更连贯了
- 但下一段规划读到的状态还没稳定
- 反而把误差带进后续阶段

所以当前选择的是：

- **缩短等待**
- **而不是完全取消等待**

### 15.4 当前建议

如果后续还想继续压停顿，正确顺序应该是：

1. 先保持四元数和抓取姿态固定不变
2. 逐步调小 `goal_wait_timeout_s`
3. 再微调 `ik_pregrasp/grasp/retreat_settle_s`
4. 每调一步都看：
   - 末端到位误差
   - 抓取成功率
   - 是否引入新的触地或漏抓

不要一口气把所有等待清零，否则很容易把“动作变快”换成“精度变差”。

## 16. 修正后的工程结论

经过后续修正，这份文档对应的最终结论应该更新为：

1. `IK -> FJT` 的执行链改造本身是成立的
2. 早期那轮“不准”的结论不是纯执行后端问题，受到了错误固定四元数的明显污染
3. 在四元数恢复正常后，FJT 才能被公平评估
4. `/go_to_pose` 的卡顿感并不只是 FJT 本身，而是：
   - 轨迹执行结束后的收敛等待
   - 阶段间显式 `settle`
5. 这些停顿已经做过一轮压缩优化，当前节奏是“更快但仍保留基本稳定性”的折中值

所以，今天如果再回头总结这次架构改动，更准确的说法应该是：

**这次不是单纯把 IK 改成 FJT，而是经历了“执行链统一、姿态修正、阶段等待压缩”三步以后，才得到现在这条真正可用的链路。**

这也是为什么在当前项目里，这一步被视为一次真正的架构修正，而不只是普通调参。
